1. Architecture Overview

This is an 8-stage pipelined RISC-V processor implementing the RV32I base integer instruction set. The pipeline splits a classic 5-stage design into 8 stages to improve clock frequency on FPGA targets by reducing the critical path length per stage. The stages are:

  1. Fetch (F) - PC generation and update
  2. Instruction Memory (IM) - Synchronous instruction memory read (1-cycle latency)
  3. Decode (D) - Instruction decoding, control signal generation, immediate extraction
  4. Register Read (RR) - Register file read (synchronous, 1-cycle latency), forwarding decision
  5. Execute 1 (EX1) - Operand selection with forwarding muxes
  6. Execute 2 (EX2) - ALU computation, branch condition evaluation, branch target calculation
  7. Execute 3 / Memory (EX3/MEM) - PC target finalization, memory read/write initiation
  8. Writeback (WB) - Write-back data selection and register file write

The processor uses a Harvard memory architecture with separate instruction and data memories. Both memories are synchronous (M9K BRAM-based), introducing one-cycle read latency each. Data hazards are resolved through a combination of stalling (hazard detection unit) and data forwarding (forwarding unit). Branch and jump resolution occurs in Stage 7 (EX3/MEM), with a flush penalty of 6 pipeline stages on taken branches and jumps.

4. Stage 1 - Fetch (F)

Module: pc_update

File: rtl/core/1_fetch/pc_update.sv

Generates the program counter. On each rising clock edge, updates PC based on priority:

  1. reset - PC resets to 32'b0
  2. pc_sel (branch/jump taken) - PC loads pc_target from Stage 7
  3. stall (hazard detected) - PC holds its current value
  4. Default - PC increments by 4 (pc + 4)

Port Interface

PortDirectionWidthDescription

resetinput1Synchronous reset
stallinput1From hazard detection unit (stall_frontend)
pc_selinput1Branch/jump taken signal from Stage 7 (E3_pc_sel)
pc_targetinput32Target PC from Stage 7 (E3_pc_target)
pcoutput32Current program counter (F_pc)

5. Stage 2 - Instruction Memory (IM)

Module: instr_mem

File: rtl/core/2_imem/instr_mem.sv

Synchronous instruction memory modeled as a 1024-entry ROM/RAM array, initialized from program.hex via $readmemh. Uses M9K BRAM inference on Intel/Altera FPGAs. The synchronous read introduces a 1-cycle latency, justifying a dedicated pipeline stage.

Behavior

  • On each clock edge (when not stalled): reads instruction_memory[pc[31:2]] into instr_reg and latches pc into pc_reg.
  • A flush_reg flag is set on reset or flush, and cleared on the next non-stalled cycle.
  • When flush_reg is active, the output instruction is replaced with NOP (32'h00000013, which is ADDI x0, x0, 0) and pc_out is zeroed.

Port Interface

PortDirectionWidthDescription

resetinput1Synchronous reset
stallinput1Stall signal (stall_frontend)
flushinput1Flush signal (E3_pc_sel)
pcinput32Program counter from Stage 1 (F_pc)
pc_outoutput32Registered PC (IM_pc_out)
instructionoutput32Fetched instruction (IM_instruction)

Memory Parameters

ParameterValueDescription
MEM_DEPTH1024Number of 32-bit instruction words (4 KiB)

6.5 Module: ID_RR (Pipeline Register)

File: rtl/core/3_decode/ID_RR.sv

Pipeline register at the Decode/Register-Read boundary. Registers all decoded data fields and control signals. On reset or flush, all outputs are zeroed (inserting a NOP bubble). On stall, holds current values.

Important: In the core instantiation, ID_RR.flush is driven by flush || stall_frontend. This means that when the frontend is stalled, the ID_RR register is flushed (bubble inserted) rather than held. This prevents a stalled instruction from being duplicated into the backend.

Registered Signals

Immediate (32b), rs1 (5b), rs2 (5b), rd (5b), pc (32b), uses_rs1 (1b), uses_rs2 (1b), alu_op (4b), alu_src_a (1b), alu_src_b (1b), reg_write (1b), mem_read (1b), mem_write (1b), mem_size (2b), mem_unsigned (1b), wb_sel (2b), branch (1b), jump (1b), branch_type (3b).

7. Stage 4 - Register Read (RR)

Stage 4 reads operands from the register file (synchronous read, 1-cycle latency) and computes forwarding select signals for Stage 5.

7.1 Module: regfile (Register File)

File: rtl/core/4_reg_read/regfile.sv

32-entry x 32-bit register file with two synchronous read ports and one synchronous write port. Register x0 is hardwired to zero (writes to x0 are suppressed). Supports read-during-write forwarding: if the write address matches a read address in the same cycle, the write data is forwarded to the read output.

Port Interface

PortDirectionWidthDescription
clkinput1System clock
read_address1input5Read port 1 address (IDRR_rs1)
read_address2input5Read port 2 address (IDRR_rs2)
read_data1output32Read port 1 data (E1_rs1_data)
read_data2output32Read port 2 data (E1_rs2_data)
write_addressinput5Write port address (W_rd)
write_datainput32Write port data (W_write_data)
write_enableinput1Write enable (W_reg_write)
Note: The register file uses synchronous reads (clocked), which means the read data appears one cycle after the address is presented. This is why the register file read occupies its own pipeline stage (Stage 4). The read addresses come from the ID_RR register (end of Stage 3), and the read data emerges at the input to Stage 5 (EX1).

7.2 Module: forwarding_unit

File: rtl/core/hazard_control/forwarding_unit.sv

Computed in Stage 4 (using ID_RR outputs) but the select signals are consumed in Stage 5 (via RR_EX1 register). See Section 13: Data Forwarding for full details.

7.3 Module: RR_EX1 (Pipeline Register)

File: rtl/core/4_reg_read/RR_EX1.sv

Pipeline register at the Register-Read/EX1 boundary. Registers all data, control, and forwarding select signals. On reset or flush, all outputs are zeroed. No stall input (this register is never stalled; only the frontend stages 1-3 stall).

Registered Signals

All signals from ID_RR, plus: forward_a_sel (2b) and forward_b_sel (2b).

8. Stage 5 - Execute 1 (EX1)

Stage 5 applies forwarding muxes to select the correct operand values before the ALU.

8.1 Module: data_sel (Operand Selection Unit)

File: rtl/core/5_ex1/data_sel.sv

Multiplexes between register file data, forwarded data from downstream stages, PC, and immediate values to produce the final ALU operands.

Operand A Selection

  1. forward_a_sel selects the base rs1 value:
    • 2'b00: rs1_data (from register file)
    • 2'b01: fwd_ex2_data (forwarded from EX2/EX3 boundary = E3_fwd_val)
    • 2'b10: fwd_ex3_data (forwarded from EX3/WB boundary = W_fwd_val)
  2. alu_src_a selects between rs1_final (0) and pc (1).

Operand B Selection

  1. forward_b_sel selects the base rs2 value (same encoding as above).
  2. alu_src_b selects between rs2_final (0) and imm (1).

Additionally, rs2_data_out outputs the forwarded rs2 value (before the alu_src_b mux), which is used as store data for memory write instructions.

Forwarding Value Sources (defined in core.sv)

SignalSourceValue
E2_fwd_valConnected to fwd_ex2_data via E3_fwd_val(E2_wb_sel == 2'b10) ? E2_pc + 4 : E2_alu_result
E3_fwd_valConnected to fwd_ex2_data input(E3_wb_sel == 2'b10) ? E3_pc + 4 : E3_alu_result
W_fwd_valConnected to fwd_ex3_data inputW_write_data
Important wiring detail: In core.sv, data_sel.fwd_ex2_data is connected to E3_fwd_val (not E2_fwd_val), and data_sel.fwd_ex3_data is connected to W_fwd_val. The third input fwd_wb_data is tied to 32'b0 and is currently unused. The forwarding select encoding from the forwarding unit (2'b01 = EX2 stage result, 2'b10 = EX3 stage result) maps to these physical connections.

9. Stage 6 - Execute 2 (EX2)

Stage 6 performs the main ALU computation and evaluates branch conditions.

9.1 Module: alu (Arithmetic Logic Unit)

File: rtl/core/6_ex2/alu.sv

Purely combinational ALU. All operations are computed in parallel; the control input selects which result to output.

Port Interface

PortDirectionWidthDescription
Ainput32Operand A (E2_operand_a)
Binput32Operand B (E2_operand_b)
controlinput4ALU operation select (E2_alu_op)
resultoutput32ALU result (E2_alu_result)

Operations

ADD, SUB, AND, OR, XOR are straightforward 32-bit operations. Shifts (SLL, SRL, SRA) use only B[4:0] as the shift amount. SLT and SLTU produce a 32-bit result with the comparison result in bit 0 and zeros in bits [31:1]. ALU_PASS outputs B directly (used for LUI).

10. Stage 7 - Execute 3 / Memory (EX3/MEM)

Stage 7 performs three operations: PC target finalization for branches/jumps, memory access initiation, and preparation of the writeback intermediate value. This is the stage where branch/jump resolution occurs, generating the flush signal.

10.1 Module: pc_target_calc (PC Target Calculator)

File: rtl/core/7_ex3_mem/pc_target_calc.sv

Computes the final PC selection signal and target address for branches and jumps.

Port Interface

PortDirectionWidthDescription
pcinput32PC of the instruction (E3_pc)
operand_ainput32Operand A (unused in current logic)
operand_binput32Operand B (unused in current logic)
branchinput1Branch flag (E3_branch)
jumpinput1Jump flag (E3_jump)
branch_typeinput3Branch type (unused in current logic)
imminput32Immediate value (E3_imm)
alu_resultinput32ALU result (E3_alu_result)
condition_met_ininput1Branch condition result (E3_condition_met)
branch_target_ininput32Branch target from Stage 6 (E3_branch_target)
pc_seloutput1PC redirect signal (E3_pc_sel)
pc_targetoutput32Target PC value (E3_pc_target)

Behavior

  • pc_sel = jump || (branch && condition_met_in)
  • If jump: pc_target = alu_result & 32'hFFFFFFFC (4-byte aligned)
  • If branch && condition_met: pc_target = branch_target_in & 32'hFFFFFFFC (4-byte aligned)
  • Otherwise: pc_target = 32'b0
JAL vs JALR target computation:
  • JAL: alu_src_a=1 (PC), alu_src_b=1 (imm). ALU computes PC + imm. pc_target_calc uses alu_result & 0xFFFFFFFC as the jump target.
  • JALR: alu_src_a=0 (rs1), alu_src_b=1 (imm). ALU computes rs1 + imm. pc_target_calc uses alu_result & 0xFFFFFFFC as the jump target, clearing the lowest 2 bits per the RISC-V specification.
  • Branch: Target is branch_target_in (= PC + imm, computed in branch_eval), also 4-byte aligned.

10.2 Module: memory (Memory System Top-Level)

File: rtl/core/7_ex3_mem/memory.sv

Memory interconnect that routes read/write requests to either data memory (BRAM) or MMIO based on the address. See Section 15: Memory System for full details.

10.3 Writeback Intermediate Value

Computed in core.sv as combinational logic:

W_wb_intermediate = (E3_wb_sel == 2'b10) ? E3_pc + 32'd4 : E3_alu_result

This pre-computes the non-memory writeback value (ALU result or PC+4 for link register writes). The value is registered through MEM_WB as alu_result_in.

10.4 Module: MEM_WB (Pipeline Register)

File: rtl/core/7_ex3_mem/MEM_WB.sv

Pipeline register at the MEM/WB boundary. On reset or flush (currently hardwired to 1'b0), all outputs are zeroed.

This means memory read data arrives one cycle later than the registered ALU result, and the writeback mux operates with this timing difference.

Stall Conditions

A stall is asserted when any of the following conditions are true (all require rd != x0):

  1. RR stage dependency: Instruction in RR (Stage 4) writes to a register that the instruction in Decode (Stage 3) reads. This covers the case where an instruction is only 1 stage ahead and its result is not yet available anywhere for forwarding.
  2. EX1 stage dependency: Instruction in EX1 (Stage 5) writes to a register that the instruction in Decode reads. This covers the 2-stage gap where the result is not yet computed.
  3. EX2 load-use hazard: Instruction in EX2 (Stage 6) is a load (E2_mem_read) and its destination matches a source register in Decode. Load data is not available until after the memory read completes.
  4. EX3 load-use hazard: Instruction in EX3 (Stage 7) is a load (E3_mem_read) and its destination matches a source register in Decode. Memory read data is still being fetched.

Stall Effect

  • stall_frontend freezes Stages 1-3: PC holds, IMEM holds, IF_ID holds, ID_RR is flushed.
  • The ID_RR flush on stall (flush || stall_frontend) inserts a NOP bubble into the backend pipeline.
  • Stages 4-8 continue to advance normally during a stall.

Module: forwarding_unit

14. Branch and Jump Handling

14.1 Resolution Point

Branches and jumps are resolved in Stage 7 (EX3/MEM) by the pc_target_calc module. The E3_pc_sel signal (= flush) indicates a taken branch or jump.

14.2 Flush Mechanism

When E3_pc_sel is asserted:

  • flush signal is broadcast to all pipeline registers from IF_ID through EX2_EX3.
  • All 6 pipeline stages between Fetch and EX3 are flushed (instructions converted to NOPs).
  • pc_update loads the target PC on the next clock edge.
  • instr_mem sets flush_reg to output a NOP on the next cycle.

14.3 Branch Penalty

Taken branches and jumps incur a 6-cycle penalty (Stages 1-6 are flushed). Not-taken branches incur no penalty (they flow through the pipeline normally).

There is no branch prediction in the current design. All branches are predicted not-taken (the PC always increments by 4 unless redirected).

20. Timing and Performance

20.1 Critical Path Candidates

  • ALU path: EX1_EX2 register outputs through ALU combinational logic to EX2_EX3 register inputs (Stage 6).
  • Forwarding path: Forwarding unit combinational logic through RR_EX1 setup time (Stage 4).
  • Hazard detection path: Decode combinational outputs through hazard detection unit to stall_frontend and back to IF_ID/ID_RR/pc_update (Stage 3).
  • Memory path: Data memory read through alignment and sign-extension logic (Stage 7 to Stage 8).

21. Corner Cases and Assumptions

21.1 Assumptions

  • All instructions are 32-bit aligned (PC[1:0] is always 00).
  • Instruction memory is word-addressed using pc[31:2]; byte addressing within instruction words is not supported.
  • No exceptions, interrupts, or traps are implemented.
  • No CSR (Control and Status Register) instructions are supported.
  • No FENCE, ECALL, or EBREAK instructions are implemented.
  • The M extension (multiply/divide) is not implemented.
  • The A extension (atomics) is not implemented.
  • Memory accesses are assumed to complete in 1 cycle (no cache misses, no wait states).
  • MMIO reads and writes are registered (1-cycle latency).
  • The register file is initialized to all zeros.
  • Program memory is loaded at synthesis time via $readmemh("program.hex").

21.2 Known Corner Cases

  • Register x0 handling: The register file suppresses writes to x0 and the hazard/forwarding units check for rd != 5'b0 and rsN != 5'b0 to avoid false dependencies on x0.
  • Simultaneous stall and flush: The pc_sel (flush) has higher priority than stall in pc_update. If a branch resolves while the frontend is stalled, the branch redirect takes precedence and the stalled instruction is discarded.
  • MEM_WB flush: The MEM_WB.flush input is hardwired to 1'b0. This means instructions that have reached Stage 7 are never flushed. This is correct because the branch/jump resolution also occurs in Stage 7, so any instruction at that stage or beyond is architecturally committed.
  • Memory read data bypass: W_mem_read_data comes directly from the memory module (which has its own internal registered output), not through the MEM_WB pipeline register (where mem_read_data_in is tied to 32'b0). This effectively means the memory data arrives at the writeback mux aligned with the MEM_WB register outputs due to the memory's internal read register.
  • NOP encoding: 32'h00000013 is ADDI x0, x0, 0, which is the standard RISC-V NOP. It writes to x0 (no effect), reads x0 (always zero), and adds zero. reg_write will be asserted for this instruction, but the register file suppresses the write to x0.
  • JALR alignment: pc_target_calc applies & 32'hFFFFFFFC to both jump and branch targets, clearing the lowest 2 bits to enforce 4-byte alignment. For JAL, the immediate is always 2-byte aligned by construction, so the mask clears at most bit 1. For JALR, the mask is required by the RISC-V specification to clear bit 0 of the computed target (rs1 + imm).
