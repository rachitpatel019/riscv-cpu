1. Architecture Overview

This is an 8-stage pipelined RISC-V processor implementing the RV32I base integer instruction set. The pipeline splits a classic 5-stage design into 8 stages to improve clock frequency on FPGA targets by reducing the critical path length per stage. The stages are:

  1. Fetch (F) - PC generation and update; PC redirected by Stage 4 branch prediction or Stage 7 branch/jump resolution
  2. Instruction Memory (IM) - Synchronous instruction memory read (1-cycle latency)
  3. Decode (D) - Instruction decoding, control signal generation, immediate extraction, and BHT read index generation
  4. Register Read (RR) - Register file read (synchronous, 1-cycle latency), forwarding decision, early branch target address calculation (PC + imm), and BHT branch prediction evaluation
  5. Execute 1 (EX1) - Operand selection with forwarding muxes
  6. Execute 2 (EX2) - ALU computation, branch condition evaluation
  7. Execute 3 / Memory (EX3/MEM) - PC target finalization, BHT update, memory read/write initiation
  8. Writeback (WB) - Writeback data selection and register file write

The processor uses a Harvard memory architecture with separate instruction and data memories. Both memories are synchronous (M9K BRAM-based), introducing one-cycle read latency each. Data hazards are resolved through a combination of stalling (hazard detection unit) and data forwarding (forwarding unit). Branch target addresses are computed early in Stage 4 (RR) using the decoded PC and immediate, eliminating the one-cycle adder from Stage 6's critical path. A Branch History Table (BHT) provides dynamic taken/not-taken prediction. The BHT is queried in Decode (Stage 3) and evaluated in Register Read (Stage 4). When a branch is predicted taken, it redirects the PC from Stage 4 (RR), reducing the taken-branch penalty from 6 cycles to 3 cycles when the prediction is correct. Unconditional JAL jumps are also resolved early in Stage 4 (RR), incurring a 3-cycle flush penalty. Final branch and JALR jump resolution occurs in Stage 7 (EX3/MEM); a branch misprediction or JALR jump incurs a 6-cycle flush penalty.

5. Stage 2 - Instruction Memory (IM)

Module: instr_mem

File: rtl/core/2_imem/instr_mem.sv

Synchronous instruction memory modeled as a 16384-entry ROM/RAM array, initialized from program.hex via $readmemh. Uses M9K BRAM inference on Intel/Altera FPGAs. The synchronous read introduces a 1-cycle latency, justifying a dedicated pipeline stage.

Design Rationale — Inferring BRAM for ROM and RAM Blocks
Standard FPGA logic cells (LEs) are built to implement logic gates and individual registers, not bulk memory arrays. Registering 64 KB of Instruction ROM and 64 KB of Data RAM using Logic Elements would completely exhaust the target device resources and cause severe place-and-route congestion. Instead, this design structures ROM and RAM modules to infer physical block RAM (M9K blocks). This maps memory accesses directly onto high-speed dedicated static memory blocks on the silicon, leaving general logic elements free for the CPU execution pipeline control logic.

Behavior

  • On each clock edge (when not stalled): reads instruction_memory[pc[31:2]] into instr_reg and latches pc into pc_reg.
  • A flush_reg flag is set on reset or flush, and cleared on the next non-stalled cycle.
  • When flush_reg is active, the output instruction is replaced with NOP (32'h00000013, which is ADDI x0, x0, 0) and pc_out is zeroed.

7. Stage 4 - Register Read (RR)

Stage 4 reads operands from the register file (synchronous read, 1-cycle latency), computes forwarding select signals for Stage 5, and calculates the branch target address (PC + sign-extended immediate) combinationally from the ID_RR register outputs.

Design Rationale — Why Evaluate Branches in Stage 4?
Evaluating branch predictions in Stage 4 (Register Read) instead of Stage 3 (Decode) is a key timing and penalty optimization. Reading the BHT is initiated in Stage 3, and the resulting prediction is evaluated in Stage 4 where register operands and PC-relative branch target addresses (calculated combinationally in core.sv) are first available. By redirecting the PC from Stage 4 (flushing only stages 1–3), we limit the taken-branch penalty to exactly 3 cycles while keeping the BHT read and target addition paths from bloat-timing the Stage 3 critical path.

7.1 Module: regfile (Register File)

File: rtl/core/4_reg_read/regfile.sv

32-entry x 32-bit register file with two synchronous read ports and one synchronous write port. To guarantee dedicated block RAM (BRAM) mapping on the FPGA target (such as Intel M9K blocks) and eliminate LUT-based soft-logic usage, the register file uses two identical replicated RAM arrays: registers_a and registers_b. Each replicated block is configured as a Simple Dual-Port (1R1W) BRAM.

Write operations update both registers_a and registers_b simultaneously to keep the BRAM states synchronized. Read port 1 (rs1) reads from registers_a, and read port 2 (rs2) reads from registers_b. Register x0 is hardwired to zero (writes to x0 are suppressed, and it always reads as zero).

Supports explicit synchronous read-during-write forwarding: if a register is written and read in the same cycle, the write data is forwarded directly to the registered read outputs on the next clock edge, avoiding read collisions within the physical BRAM cells.

7.1.1 Register File Organization & Replicated BRAMs

The register file must support two concurrent read ports (rs1, rs2) and one write port (rd) in a single cycle. However, physical block RAM structures on Intel FPGAs (M9K cells) are dual-ported (supporting only 1 Read/1 Write or 2 Read/2 Write under restricted configurations).

To implement 2 Read ports and 1 Write port without using inefficient register logic (LE-based flip-flops), the Register File is designed as Replicated BRAMs:

  • Replication: The 32x32 register array is duplicated into two identical physical RAMs: registers_a and registers_b.
  • Write Operations: When a writeback instruction commits in Stage 8, both registers_a and registers_b are written simultaneously at the same write_address with the same write_data. This keeps their contents completely identical.
  • Read Operations: Read port 1 (rs1) reads from registers_a, and Read port 2 (rs2) reads from registers_b. Since they access independent RAM blocks, both reads execute simultaneously in a single cycle.
  • Collision Handling: Read-during-write conflicts (accessing a register being written in the same clock cycle) are resolved by local forwarding logic inside the regfile module. If the read index matches the write index, the write data is returned directly instead of waiting for the internal BRAM array update.

7.3 Branch Target Calculation

The branch target address is calculated combinational in core.sv from the ID_RR register outputs: assign IDRR_branch_target = IDRR_pc + IDRR_immediate;. The result is registered through RR_EX1 and EX1_EX2 and consumed by pc_target_calc in Stage 7. This early calculation removes the branch-target adder from Stage 6's critical path.

Scope: This calculation computes the B-type and J-type (JAL) PC-relative target only. JALR targets depend on a register value (rs1 + imm), which is not available until EX1; the JALR target continues to be computed by the ALU in Stage 6 and finalized by pc_target_calc in Stage 7.

9. Stage 6 - Execute 2 (EX2)

Stage 6 performs the main ALU computation and evaluates branch conditions. Branch target calculation has been moved to Stage 4; the pre-computed target arrives via the EX1_EX2 pipeline register and is passed through to Stage 7 without recomputation.

Design Rationale — Splitting the Execute Logic into 3 Stages
In standard 5-stage pipelines, the Execute stage is the primary critical path because it performs operand forwarding multiplexing, ALU calculations, and branch target resolution in a single clock cycle. To maximize clock frequency ($F_{\text{max}}$) on the MAX 10 FPGA, this design divides these operations across three pipelined stages: operand selection (EX1), ALU computation (EX2), and branch resolution/memory routing (EX3). This split successfully increased the maximum clock frequency to 127.29 MHz.

Stall Conditions

A stall is asserted when any of the following conditions are true (all require rd != x0):

  1. RR stage dependency: Instruction in RR (Stage 4) writes to a register that the instruction in Decode (Stage 3) reads. This covers the case where an instruction is only 1 stage ahead and its result is not yet available anywhere for forwarding.
  2. EX1 stage dependency: Instruction in EX1 (Stage 5) writes to a register that the instruction in Decode reads. This covers the 2-stage gap where the result is not yet computed.

Stall Effect

  • stall_frontend freezes Stages 1-3: PC holds, IMEM holds, IF_ID holds, ID_RR is flushed.
  • The ID_RR flush on stall (flush || stall_frontend) inserts a NOP bubble into the backend pipeline.
  • Stages 4-8 continue to advance normally during a stall.

14. Branch and Jump Handling

14.1 Resolution Point

Unconditional JAL jumps are resolved early in Stage 4 (RR) by the stage4_pc_sel logic using the early branch target calculator. Conditional branches and JALR jumps are resolved in Stage 7 (EX3/MEM) by the pc_target_calc module. The redirect signal (stage4_pc_sel in Stage 4 or E3_pc_sel in Stage 7) triggers a pipeline flush.

14.2 Flush Mechanism

When a redirect is asserted:

  • For early JAL resolution in Stage 4: the stage4_flush signal flushes Stages 2 and 3 (IMEM and IF_ID).
  • For Stage 7 branch/JALR resolution: the flush signal (driven by E3_pc_sel) is broadcast to all pipeline registers from IF_ID through EX2_EX3, clearing all 6 in-flight stages.
  • The pc_update module loads the correct target PC on the next clock edge.
  • The BHT entry for the resolved branch is updated with the actual outcome.

14.3 Branch and Jump Penalty

Unconditional jumps and conditional branches incur the following penalties:

  • Unconditional JAL Jump: 3-cycle flush penalty (resolved in Stage 4).
  • Unconditional JALR Jump: 6-cycle flush penalty (resolved in Stage 7).
  • Conditional Branch (correct prediction): 0 extra cycles (for predicted not-taken) or 3 cycles (for predicted taken, redirected in Stage 4).
  • Conditional Branch (misprediction): 6-cycle flush penalty (redirected in Stage 7).

Not-taken branches that are correctly predicted incur no penalty. The BHT reduces the average branch penalty compared to the prior always-predict-not-taken scheme.

The BHT uses standard 2-bit saturating counters. The state transitions are shown below:

Prediction Evaluation and Usage (Stage 4 - Register Read)

The synchronous BHT read outputs the counter value BRAM_counter_out in Stage 4 (Register Read) as IDRR_counter_val. The prediction is evaluated combinational: assign IDRR_predict_taken = IDRR_branch && (IDRR_counter_val[1] == 1'b1);. If predicted taken, PC redirection is triggered via: assign stage4_pc_sel = IDRR_predict_taken || IDRR_is_jal;, which loads stage4_pc_target = IDRR_branch_target (pre-computed IDRR_pc + IDRR_immediate) into the PC register.

The updated priority for PC selection in pc_update (Stage 1) is:

  1. reset - PC resets to 32'b0
  2. pc_sel (branch/jump resolved in Stage 7, or misprediction correction) - PC loads Stage 7 target E3_pc_target
  3. stage4_pc_sel (BHT predicts taken or early JAL in Stage 4) - PC loads Stage 4 target IDRR_branch_target
  4. stall (hazard detected) - PC holds its current value
  5. Default - PC increments by 4 (pc + 4)
Prediction Penalty: Because the BHT prediction is evaluated in Stage 4, redirecting a predicted-taken branch flushes 3 in-flight instructions (in Stage 1, Stage 2, and Stage 3). A correctly predicted taken branch therefore incurs a 3-cycle penalty. A correctly predicted not-taken branch incurs 0 penalty. Branch mispredictions (either predicted taken but actually not-taken, or predicted not-taken but actually taken) are resolved in Stage 7 and incur a 6-cycle penalty.

15.2 BHT Timing and Compatibility Notes

  • The BHT is mapped to M9K block RAM blocks on the FPGA target, ensuring low logic utilization and meeting target frequency constraints.
  • The BHT does not predict jump targets for JALR; JALR is always resolved in Stage 7 and flushes the pipeline. JAL is resolved early in Stage 4.
  • The BHT does not store branch target addresses; it only stores taken/not-taken history. The actual target address comes from the combinational PC+imm calculation in Stage 4.
  • No BHT index aliasing protection (e.g., tags) is implemented. Different PCs mapping to the same index may interfere. This is an accepted limitation for the area budget.

21. Timing and Performance

21.1 Critical Path Candidates

  • ALU path: EX1_EX2 register outputs through ALU combinational logic to EX2_EX3 register inputs (Stage 6). Stage 6 no longer contains the branch-target adder, reducing its combinational depth slightly.
  • Branch target path (Stage 4): ID_RR register outputs through the combinational adder to RR_EX1 inputs. This path is added in Stage 4; the addition (32-bit ripple or fast-carry) must complete within the Stage 4 clock period.
  • Forwarding path: Forwarding unit combinational logic through RR_EX1 setup time (Stage 4).
  • Hazard detection path: Decode combinational outputs through hazard detection unit to stall_frontend and back to IF_ID/ID_RR/pc_update (Stage 3).
  • Memory path: Data memory read through alignment and sign-extension logic (Stage 7 to Stage 8).
  • Branch resolution path: pc_target_calc combinational logic feeding back to pc_update (Stage 7 to Stage 1).
  • BHT prediction path: BHT synchronous read data output in Stage 4 to pc_update input (Stage 1). Since BHT is read synchronously, the read output (BRAM counter value) is available in Stage 4 and is used to compute the prediction combinational in Stage 4. This prediction path feeds back to the Stage 1 PC update register combinational, which is a cross-stage path that must complete within the Stage 4 clock period.

22. Corner Cases and Assumptions

22.1 Assumptions

  • All instructions are 32-bit aligned (PC[1:0] is always 00).
  • Instruction memory is word-addressed using pc[31:2]; byte addressing within instruction words is not supported.
  • No exceptions, interrupts, or traps are implemented.
  • No CSR (Control and Status Register) instructions are supported.
  • No FENCE, ECALL, or EBREAK instructions are implemented.
  • The M extension (multiply/divide) is not implemented.
  • The A extension (atomics) is not implemented.
  • Memory accesses are assumed to complete in 1 cycle (no cache misses, no wait states).
  • MMIO reads and writes are registered (1-cycle latency).
  • Program memory is loaded at synthesis time via $readmemh("program.hex").

22.2 Known Corner Cases

  • Register x0 handling: The register file suppresses writes to x0 and the hazard/forwarding units check for rd != 5'b0 and rsN != 5'b0 to avoid false dependencies on x0.
  • Simultaneous stall and flush: The pc_sel (flush) has higher priority than stall in pc_update. If a branch resolves while the frontend is stalled, the branch redirect takes precedence and the stalled instruction is discarded.
  • Memory read data alignment: W_mem_read_data is registered through the MEM_WB pipeline register (from W_mem_read_data_raw), aligning it perfectly with the other registered writeback signals (such as W_alu_result) for Stage 8 writeback selection.
  • NOP encoding: 32'h00000013 is ADDI x0, x0, 0, which is the standard RISC-V NOP. It writes to x0 (no effect), reads x0 (always zero), and adds zero. reg_write will be asserted for this instruction, but the register file suppresses the write to x0.
  • JALR alignment: pc_target_calc applies & 32'hFFFFFFFC to both jump and branch targets, clearing the lowest 2 bits to enforce 4-byte alignment. For JAL, the immediate is always 2-byte aligned by construction, so the mask clears at most bit 1. For JALR, the mask is required by the RISC-V specification to clear bit 0 of the computed target (rs1 + imm).

22.3 Branch Target and BHT Corner Cases

  • JAL early resolution: JAL is resolved in Stage 4 (RR) and its target redirects the PC, flushing Stages 2-3. When JAL reaches Stage 7, its jump control is filtered to prevent double redirection, while JALR still redirects normally in Stage 7.
  • JALR branch target: The early branch target calculation in Stage 4 only covers PC-relative targets (B-type, JAL). JALR targets require rs1 which is not available until Stage 5 (after forwarding). The JALR target is still computed by the ALU in Stage 6 and JALR is resolved in Stage 7, incurring a 6-cycle flush penalty.
  • BHT index aliasing: Two branch instructions whose PCs alias to the same BHT index may corrupt each other's counter state. No tag bits are stored. This is a known limitation.
  • Simultaneous BHT update and read: If the update index (from Stage 7) aliases the read index (from Stage 3) in the same cycle, the write takes priority on the following cycle. The current read returns the old counter value.
  • Branch instruction in stall: When stall_frontend is asserted, the Decode PC holds and the BHT lookup is re-presented with the same PC on successive cycles. No spurious update occurs because BHT update is gated by E3_branch, which is independent of the stall signal.
  • Reset state: All BHT counters reset to Weakly Not-Taken (2'b01), providing a conservative not-taken default consistent with the prior predict-not-taken baseline.

23. Glossary of FPGA Terminology

The following glossary defines core hardware and FPGA concepts referenced throughout this specification:

  • BRAM (Block RAM): Dedicated, dual-port static memory arrays embedded directly inside the FPGA silicon (such as Intel's M9K blocks). Unlike logic-cell registers, BRAM provides highly efficient single-cycle read and write access, allowing implementation of large memory blocks (Instruction/Data Memory, Register File, and Branch History Table) without depleting the FPGA's general logic cells.
  • Logic Element (LE): The standard unit of logic in Intel FPGAs. A logic element contains a Look-Up Table (LUT) to implement combinational logic gates, a programmable register (D flip-flop) to store data, and dedicated carry chains for fast additions.
  • Fmax (Maximum Frequency): The maximum speed at which a digital design can run on the target FPGA without setup timing violations. In this CPU design, Fmax is optimized to 127.29 MHz by splitting logic into 8 balanced pipeline stages.
  • STA (Static Timing Analysis): A method of analyzing and checking the timing of a digital circuit by computing propagation delays along all logic paths, verifying that setup and hold times are satisfied under worst-case temperature and voltage parameters.
  • SDC (Synopsys Design Constraints): A configuration file defining timing requirements, clock constraints, input/output delays, and multi-cycle paths to guide the synthesizer and place-and-route tools during FPGA compilation.
  • JTAG (Joint Test Action Group): A standard interface and protocol used to program, test, and debug hardware devices. On the DE10-Lite board, JTAG is used by the USB-Blaster cable to upload compiled bitstreams.
  • MMIO (Memory-Mapped Input/Output): A hardware design style where external peripherals (LEDs, switches, hex displays) are assigned specific memory addresses, allowing soft program execution to perform reading or writing operations on hardware using standard load and store instructions.
  • BHT (Branch History Table): A dynamic branch prediction buffer containing 2-bit saturating counters indexed by instruction PC bits. It dynamically updates prediction directions (Taken/Not-Taken) based on execution outcomes.
  • Setup Slack: The timing margin between the clock period and the longest signal propagation time (data path delay) to a register input. A positive slack indicates that the design safely meets the timing requirements; negative slack means timing is violated.
  • Place & Route (Fitting): The compilation phase where Quartus maps logical gates to physical Logic Elements on the chip and sets physical wire routing paths between them.
